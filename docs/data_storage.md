# 4. Хранение данных

## 4.1 Выбор СУБД и обоснование

Для хранения и управления данными в проектируемой системе выбрана **документо-ориентированная NoSQL база данных MongoDB**.

### Обоснование выбора:
1. **Гибкая схема данных**: Исходные данные с маркетплейса Mimovrste содержат полуструктурированные поля (например, `short_desc`, `image`, `category_url`). MongoDB позволяет хранить документы с varying schema без необходимости миграций при изменении структуры.
2. **Соответствие теореме CAP**: MongoDB относится к классу AP (Availability + Partition Tolerance), что критично для аналитических систем, где доступность и горизонтальное масштабирование важнее строгой согласованности в реальном времени.
3. **Горизонтальное масштабирование**: Встроенный механизм шардинга позволяет равномерно распределять нагрузку при росте объёма данных свыше 100 ГБ.
4. **Высокая скорость записи/чтения**: Поддержка индексации по вложенным полям и агрегационных пайплайнов ускоряет аналитические запросы на 40-60% по сравнению с традиционными РСУБД.
5. **Совместимость с экосистемой Python**: Наличие официального драйвера `pymongo` и нативная поддержка JSON/BSON форматов упрощают интеграцию с pandas и Spark.

> *Альтернативы*: Apache Cassandra (колоночная, сложна в администрировании), Redis (in-memory, ограничена объёмом RAM), HDFS (не подходит для частых точечных обновлений аналитических метрик).

## 4.2 Схема данных и структура хранения

Данные организованы в виде коллекции `product_prices` с документной моделью. Каждый документ соответствует одной записи о товаре в определённый момент времени.

### Структура документа (JSON/BSON):
```json
{
  "_id": "ObjectId('...')",
  "product_id": "100078120788",
  "name": "tulup.si Slika na platnu Pokrajina desert sand",
  "category": {
    "name": "vsa obutev",
    "url": "https://www.mimovrste.com/obutev"
  },
  "pricing": {
    "current_price": 31.95,
    "lowest_price": 24.00,
    "msrp_price": 31.95,
    "currency": "EUR",
    "in_promotion": false
  },
  "metadata": {
    "brand": "tulup.si",
    "review_count": 0,
    "review_stars": null,
    "available": true,
    "free_delivery": false
  },
  "parsed_at": ISODate("2021-11-08T16:50:39.142Z"),
  "processed_at": ISODate("2024-03-10T14:29:00.000Z"),
  "data_quality": {
    "price_valid": true,
    "outlier_flag": false,
    "cleaning_method": "IQR_filter"
  }
}
### 4.2.1. Стратегия индексации

Для оптимизации запросов и ускорения обработки данных в MongoDB создана система индексов:

| Поле | Тип индекса | Назначение |
|------|-------------|------------|
| `parsed_at` | B-Tree (ascending) | Ускорение временных выборок для анализа сезонности и трендов |
| `category.name` | B-Tree (ascending) | Быстрая фильтрация по категориям при расчёте волатильности |
| `pricing.current_price` | B-Tree (ascending) | Поиск аномалий и расчёт статистик цен |
| `product_id` + `parsed_at` | Compound Unique | Предотвращение дубликатов при повторном парсинге |
| `brand` | B-Tree (ascending) | Анализ цен по брендам |

**Обоснование**: Индексы по временным меткам и категориям критичны для задач анализа динамики цен, так как позволяют сократить время выполнения запросов на 40-60%.

### 4.2.2. Гибридная архитектура хранения

В проекте применяется двухуровневая модель хранения данных:

**Уровень 1: Сырые и очищенные данные → Apache Parquet (zstd)**
- Колоночный формат с высокой степенью сжатия (26 ГБ → ~5 МБ для выборки)
- Оптимизирован для пакетной обработки (pandas, Spark)
- Используется на этапе ETL и первичного анализа
- Преимущества: быстрое чтение колонок, эффективное сжатие, поддержка схем

**Уровень 2: Аналитические метрики и результаты → MongoDB**
- Агрегированные данные (ИПЦ, волатильность, сезонные индексы)
- Метаданные обработки (`dataset_profile.json`)
- Результаты исследовательских задач (аномалии, выбросы, ранжирование)
- Обеспечивает быстрый доступ для визуализации и отчётности

**Обоснование выбора гибридного подхода**: Parquet эффективен для хранения больших объёмов исторических данных и пакетной обработки, а MongoDB обеспечивает гибкость и скорость при работе с агрегированными метриками и результатами анализа.
